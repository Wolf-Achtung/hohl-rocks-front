# Re-Audit hohl.rocks — Frontend + Cross-Repo

**Datum:** 2026-07-07
**Stand:** `main` nach PR #66 („Phase 2: consolidate frontend, remove dead code, fix cross-origin API calls")
**Umfang:** Vollständiger frischer Audit von `hohl-rocks-front` (alle Dateien in `public/`, netlify.toml, package.json, Root-Dokumente) plus Cross-Repo-Abgleich mit `hohl-rocks-back` (v2.8.0). Der Backend-Audit liegt als eigener Bericht im Backend-Repo (`RE_AUDIT_2026-07-07.md` in `hohl-rocks-back`).

---

## Executive Summary

Phase 2 hat die Architektur richtig gestellt: Alle API-Calls laufen same-origin über den Netlify-Proxy, das Markdown-Rendering escapt korrekt, es sind keine Secrets im Frontend, die Security-Header-Basis (HSTS, XFO, COOP, Permissions-Policy, CSP mit engem `connect-src`) ist solide, und die Video-Performance ist gut gelöst (Poster-Preload, IntersectionObserver, reduced-motion).

**Die Restprobleme sind Liegengebliebenes, kein Architekturversagen.** Der dringendste Fund ist das **Kontaktformular, das ins Leere sendet, aber Erfolg anzeigt (K1)** — Silent Data Loss auf dem einzigen Conversion-Pfad der Seite. Danach: doppelter Sound-Handler (zwei parallele Audio-Streams), clientseitiger System-Prompt (offener LLM-Proxy), eine unescapte `innerHTML`-Stelle und eine zu aggressive Cache-Policy, die Hotfixes bis zu 7 Tage verzögert.

Cross-Repo fällt auf: **Das Frontend nutzt nur noch 2 der ~12 öffentlichen Backend-Routen** (`/api/chat`, `/api/model-battle` plus `/health`). Der Rest (Prompts, Daily Challenge, News, Spark, GDPR-Self-Service) ist verwaister Backend-Code ohne Consumer — teilweise mit eigenen Sicherheits-/Kostenrisiken (siehe Backend-Bericht).

## Befunde Frontend

### KRITISCH

**K1 — Kontaktformular sendet ins Leere, zeigt aber Erfolg an**
`public/index.html:3219` (Form) + `:3877` (Submit-Handler)
Das Formular `name="kontakt"` hat **kein `data-netlify="true"`-Attribut** → Netlify registriert es beim Deploy nicht. Der JS-Handler POSTet an `/`; durch das SPA-Rewrite `/* → /index.html` (netlify.toml:35) kommt `200` zurück → `response.ok` → Erfolgsmeldung „Nachricht gesendet", obwohl nichts zugestellt wird. Der mailto-Fallback greift nur bei non-ok.
**Fix:** `data-netlify="true"` + `netlify-honeypot="bot-field"` am `<form>`; Erfolg nur bei echter Netlify-Form-Response werten; Submission nach Deploy real testen.

### HOCH

**H1 — Doppelter Click-Handler auf dem Sound-Button → zwei Audio-Streams parallel**
`public/index.html:4261` (`footerMusicToggle` → `togglePlay()`, spielt `#bg-music`) und `:4619–4627` (zweiter Listener auf dasselbe `#sound-float`, ruft `AudioController.toggle()` aus `js/audio.js` mit **eigener** `Audio('road_trip.mp3')`-Instanz). Ein Klick startet beide → Echo/Doppelton, Zustände laufen auseinander.
**Fix:** Einen Handler + `js/audio.js` komplett entfernen (redundant zum Music-Studio-Code).

**H2 — System-Prompt kommt vom Client**
`public/index.html:4507–4531` (Persona-/„Sicherheitsrichtlinien"-Prompt) + `:4543` (wird als `messages[0]` mit `role: "system"` an `/api/chat` gesendet). Jeder kann per curl denselben Endpoint mit beliebigem System-Prompt nutzen → offener LLM-Proxy auf Betreiberkosten; die „STRIKT EINHALTEN"-Regeln sind rein clientseitig.
**Fix:** System-Prompt ins Backend verlagern; Backend verwirft client-gelieferte `system`-Messages (siehe Backend-Bericht, Befund B7).

**H3 — XSS-Vektor: `result.error` unescaped in `innerHTML`**
`public/model-battle.js:377`: Fehlertexte vom Backend werden unescaped in `innerHTML` interpoliert. Da die CSP `script-src 'unsafe-inline'` erlaubt, würde injiziertes Markup ausgeführt. Verschärfend: Das Backend leakt in einigen Routen Upstream-Fehlermeldungen ungefiltert (Backend-Befund B9).
**Fix:** Per `textContent` setzen oder vor Interpolation escapen (wie `renderMarkdown` in index.html es korrekt macht).

**H4 — HTML 24h gecacht, JS 7 Tage „immutable" ohne Fingerprinting**
`public/_headers:13` (`/*` → `max-age=86400` trifft auch `index.html`) und `:22–24` (`/js/*` → `max-age=604800, immutable` für unversionierte Dateien). Deploys/Hotfixes werden bis zu 1 Tag (HTML) bzw. 7 Tage (JS) nicht sichtbar — bei einer Site, deren gesamte Logik in index.html liegt, gravierend.
**Fix:** HTML: `Cache-Control: public, max-age=0, must-revalidate`; `immutable` nur für gehashte Assets.

### MITTEL

**M1 — Toter Link `/datenschutz` + `target="_blank"` ohne `rel="noopener"`** — `public/index.html:3041`. Route existiert nicht, SPA-Fallback liefert die Startseite. Fix: auf Impressum-Overlay umstellen bzw. Anker bereitstellen; `rel="noopener"` ergänzen.

**M2 — Skip-Link zeigt auf nicht existentes `#main`** — `public/index.html:2756`; es gibt auch kein `<main>`-Element. Fix: `<main id="main">` einführen.

**M3 — Kein `<h1>` auf der Startseite** — Überschriften beginnen bei h2 (`:2846`). SEO-/Outline-Schwäche. Fix: Manifest-Opening oder „Wolf Hohl" als h1.

**M4 — Toter JS/CSS-Code für nicht existentes Markup** — Tagline-Ticker (`:3523–3551`, kein `.tagline-item`), Portfolio-Filter (`:3496–3518`, kein `.portfolio-filter`), Hero-CSS (`:186–365`, kein Hero-Markup); `public/styles.css` besteht zu ~80 % aus Bubble-/Nav-/Tips-Styles der alten Site (genutzt wird nur `.tip-detail`). Fix: entfernen/reduzieren.

**M5 — Tote Dateien im Deploy (~25 MB)** — Nirgends referenziert: `videos/road.mp4` (11,7 MB), `road_540p_bg.*`, `road_720p_bg.*` (~11 MB), `data/{bubbles,bubble-content,news,prompts}.json`, `partials/nav.html`, `prompts/clay_animations.*.md`, `audio/README.txt` (beschreibt veralteten Stand). Fix: separater Cleanup-PR.

**M6 — api-config.js: Proxy auf nie existierendes `api.js`** — `public/js/api-config.js:128–158`: alle `window.API.*`-Methoden (chat, modelBattle, getMyData, …) warten auf `window.api` aus einem `api.js`, das nicht existiert → werfen immer; kein Code ruft sie auf. `timeout: 5000` (`:87`) ist keine gültige fetch-Option. Fix: Datei auf `debugLog`/`API.base()`/Healthcheck eindampfen; Timeout via `AbortSignal.timeout(5000)`.

**M7 — package.json beschreibt ein nicht existierendes Server-Projekt** — `start`/`dev` → `node server/index.js` (kein `server/`); Dependencies express/helmet/cors/morgan/compression ungenutzt; `lint`-Script ist ein Echo-Platzhalter. Fix: Scripts/Deps bereinigen, echtes Lint hinterlegen.

**M8 — Redirects doppelt gepflegt** — `public/_redirects` und `netlify.toml` definieren identische Proxy-Regeln; SPA-Fallback nur in netlify.toml; `_redirects` gewinnt bei Konflikten. Fix: eine Quelle (Empfehlung: netlify.toml, `_redirects` löschen).

**M9 — Model-Battle initialisiert nicht unter extensionless URL** — `public/model-battle.js:57`: prüft `pathname.includes('model-battle.html')` — bei `/model-battle` (Pretty URLs) bleibt die Seite tot. Fix: `includes('model-battle')`.

**M10 — Player spielt falschen Track / UI-Mismatch** — `#bg-music` lädt `road_trip.mp3`, Playlist kennt nur „Ambient Dreams" (`ambient_loop.mp3`); `loadTrack(0)` wird initial nie aufgerufen; Track-Counter hartkodiert „1 / 4" bei 1 Track. Fix: `loadTrack(0)` bei Init; Counter dynamisch.

**M11 — Orphan Pages** — `sitemap.xml` listet `model-battle.html` (priority 0.9) und 12 `tips/*.html`, die von keiner Seite intern verlinkt sind. Fix: intern verlinken oder aus der Sitemap nehmen.

**M12 — Accessibility-Lücken** — `#music-player` und `.track-item`s sind Divs mit Click-Handler ohne `role`/`tabindex`/Tastatur; Overlays haben `role="dialog"` ohne `aria-modal`/Focus-Trap; Chat-/Battle-Antwortcontainer ohne `aria-live`. Fix: Buttons statt Divs, `aria-modal` + Fokusmanagement, `aria-live="polite"`.

**M13 — CSP erlaubt `'unsafe-inline'` für Scripts** — `public/_headers:14`. Durch die Inline-Architektur erzwungen, hebelt den XSS-Schutz weitgehend aus (relevant i. V. m. H3). Fix: mittelfristig JS auslagern bzw. Nonces; kurzfristig H3 fixen.

**M14 — audio.js lädt 2,9 MB MP3 bei jedem Seitenaufruf** — `preload='auto'` auf Modulebene. Entfällt mit H1 (Datei löschen).

## Cross-Repo-Befunde (Frontend ↔ Backend)

**X1 — Nur 2 von ~12 Backend-Routen haben einen Frontend-Consumer.**
Das Frontend ruft ausschließlich `/api/chat` und `/api/model-battle` auf (plus `/health` via Proxy). Verwaist: `/api/prompt-generator`, `/api/prompt-optimizer`, `/api/prompts`, `/api/prompts/:id`, `/api/daily-challenge`, `/api/submit-challenge`, `/api/news`, `/api/spark/today`, `/api/my-data` (GET/DELETE), `/api/self`. Zwei davon (`daily-challenge`, `submit-challenge`) sind ungeschützte Claude-Endpunkte = offener Kosten-Hahn (Backend-Befund B2). **Entscheidung nötig:** Features im Frontend reaktivieren oder Backend-Routen entfernen.

**X2 — GDPR-Lücke durch entkoppelte Features.** Das Backend loggt Chats (Message, IP, User-Agent) in Postgres, das Frontend bietet aber keinen Zugang zu den `/api/my-data`-Endpunkten (Auskunft/Löschung) — und das Session-Cookie läuft nach 24 h ab, danach sind die Daten für den Nutzer nicht mehr löschbar (Backend-Befund B14). Datenschutzerklärung + Self-Service-UI oder Lösch-Job nötig.

**X3 — Offene Frage aus `BRIEFING_BACKEND_2026-07-01.md` beantwortet:** Der Backend-Code liest **nur `ALLOWED_ORIGINS`**; die Railway-Variable `CORS_ALLOWLIST` ist tote Konfiguration und sollte in Railway gelöscht werden (Drift-Gefahr). Die Netlify-Preview-Domain (`steady-pixie-8f36d7.netlify.app`) steht nur in der toten Variable — falls Previews direkt gegen Railway sprechen sollen, muss sie in `ALLOWED_ORIGINS` umziehen. Da das Frontend inzwischen aber komplett über den Netlify-Proxy geht (same-origin), ist das Cookie-/SameSite-Problem aus dem Briefing praktisch entschärft.

**X4 — Modellnamen-Drift.** Frontend (model-battle.html:35) hartkodiert „Claude Sonnet 4, GPT-4o Mini, …", Backend nutzt per Default `claude-sonnet-5`, `gpt-5-mini`, `sonar-pro`, `gemini-3.5-flash` (env-überschreibbar) und hartkodiert wiederum eigene Anzeigenamen in `BATTLE_MODELS`. Drei Wahrheiten. Fix: Backend liefert Anzeigenamen (z. B. via `/api/self`), Frontend rendert sie.
